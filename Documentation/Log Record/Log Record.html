Project Metadata

Project Metadata window displays the administrative metadata associated with the current experiment.

Experiment metadata selection

The  Labactor software dashboard is automatically filled with experiment metadata based on the experiment schedule. Users have the opportunity to review the accuracy of this information before the Data Acquisition (DAQ) station begins capturing data, as the metadata will be embedded in every data file.

If the metadata does not match the experiment, click the ‘Change’ button to open a new window with the list of the experiments, locate and highlight your experiment, click ‘Select’ button, and close the list window.

Probe metadata can be changed via ‘Change probe’ button in a similar manner.

Open Science Framework synchronization

To enhance data management efficiency for researchers, we have automated the synchronization process of user experimental data and metadata to the Open Science Framework (OSF) repository located at osf.io. To enable synchronizing the experiment data with OSF  click the ‘Log on’ button. Upon clicking, a web browser opens prompting the user to log in with their OSF credentials. To enable our server  to write data to OSF via the OAuth2 authorization protocol click the  “Allow“ button, see below.

The software automatically generates and fills in the OSF project description and project wiki with the  proposal  metadata from the MagLab User Portal , see example below. Each Data Acquisition Station (DAQ) streams files containing experiment data to the OSF server through standard HTTPS protocol operations such as HTTPS POST, PUT, and GET requests, directed to corresponding OSF endpoints using the user's authorization tokens. For each uploaded TDMS binary file, the software  creates a companion ASCII markup file in markdown format which lists the metadata extracted from the TDMS file to enhance the data findability.

Additionally, the software populates the OSF wiki as an electronic lab book with experiment descriptions and identifiers, updating periodically with file information as data is recorded to OSF. The initial OSF project is private and can be  shared with project collaborators. The user can publicize the project and create its persistent digital object identifier (DOI).

Storage Management -  Folders and Components

By default, the software generates a new folder for each day of an experiment under the corresponding experiment component within the OSF  project. The capacity of private projects and components using OSF Storage is limited to 5 GB per each OSF component. By default, the dashboard software saves files to OSF storage. If an experiment generates large files, there is  an ‘OSF-split’ option in the ‘Storage location’  selector to create a new component under the experiment component as a workaround to utilize more storage space from OSF.

Users have the option to enhance storage management by adding OSF add-ons to their projects. They can manually link their Google Drive, Box, Dropbox, etc., accounts to their OSF account.

To write files to add-on storage instead of the native OSF storage, users can select the corresponding option from the drop-down list in the ‘Storage location’ selector.

Sharing OSF projects with Collaborators

If a user wishes to share a project and its experiments with other collaborators, this can be accomplished via ‘Contributors’ menu. We recommend selecting admin roles at the top project component level and checking all the sub-components when adding a contributor. In doing so, all newly created sub-components will automatically be visible to collaborators.
